# 1-2. 동적 API에도 CDN을 왜 쓰는가?

정적 파일은 CDN과 잘 어울립니다. 이미지, JavaScript, CSS, 동영상처럼 여러 사용자에게 같은 파일을 줄 수 있기 때문입니다.

그런데 **사용자마다 응답이 달라지는 동적 API**도 CDN 앞에 두는 경우가 많습니다. 이유는 CDN이 단순한 파일 캐시가 아니라 **엣지 서버 기반의 네트워크 입구**로 동작하기 때문입니다.

## 캐시하지 않아도 빨라질 수 있다

동적 API 응답은 매번 원본 서버까지 가야 할 수 있습니다. 그래도 CDN을 거치면 다음 최적화를 얻을 수 있습니다.

- 사용자는 가까운 엣지 서버까지만 먼저 연결합니다.
- 엣지 서버는 원본 서버와의 연결을 재사용하거나 더 안정적인 네트워크 경로를 사용합니다.
- TLS 종료, 압축, HTTP/2 또는 HTTP/3 처리 같은 연결 비용을 엣지에서 줄일 수 있습니다.
- 전 세계 사용자의 요청이 곧바로 원본으로 몰리지 않고, CDN 네트워크를 통해 정리되어 들어갑니다.

즉, 캐시 히트가 없어도 엣지는 **가까운 접속 지점**, **최적화된 프록시**, **원본 보호막** 역할을 합니다.

## 일부만 캐시할 수 있다

동적 API라고 해서 항상 캐시가 불가능한 것은 아닙니다.

- 모든 사용자에게 같은 공개 데이터
- 짧은 시간 동안만 유효한 목록 응답
- 권한과 무관한 메타데이터
- 에러 상황에서 잠깐 사용할 수 있는 stale 응답

이런 응답은 `Cache-Control`, `s-maxage`, `stale-while-revalidate` 같은 정책으로 CDN에 짧게 저장할 수 있습니다.

## 보안과 트래픽 제어가 가능하다

CDN은 API 앞단에서 보안 계층으로도 쓰입니다.

- DDoS 트래픽을 원본 서버 앞에서 흡수합니다.
- WAF로 비정상 요청을 차단합니다.
- 봇, 스크래핑, 비정상적인 반복 요청을 제한합니다.
- rate limit으로 특정 사용자나 IP의 요청 폭주를 제어합니다.
- 엣지에서 인증 헤더, 라우팅, 리다이렉트 같은 가벼운 로직을 처리할 수 있습니다.

## 정리

동적 API에서 CDN을 쓰는 핵심 이유는 **캐시 때문만이 아닙니다**.

CDN의 엣지 서버는 사용자에게 가까운 네트워크 입구가 되고, 원본 서버 앞에서 성능 최적화, 보안, 트래픽 제어, 장애 완충 역할을 합니다. 그래서 개인화된 API처럼 캐시가 제한적인 요청도 CDN 앞에 두는 것이 실무적으로 의미가 있습니다.
